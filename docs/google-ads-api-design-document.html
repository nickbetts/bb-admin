1 Application Overview

i3Media is a digital marketing agency that manages Google Ads campaigns on behalf of multiple clients under a Google Ads Manager (MCC) account. The i3Media Client Reporting Tool is an internal, private web application used exclusively by i3Media staff to pull live campaign performance data from the Google Ads API and present it in a unified reporting dashboard alongside other marketing channels (Google Analytics 4, Meta Ads, and SEMrush).

The tool aggregates key PPC metrics — clicks, cost, conversions, conversion value, ROAS, CTR, and CPA — and displays them with campaign-level and ad group-level breakdowns so account managers can monitor performance and prepare client reports efficiently.

Intended Use: Read-only reporting. The tool does not create, modify, pause, or delete any Google Ads campaigns, ad groups, ads, or keywords. All API calls are GoogleAdsService.Search (GAQL) queries only.

3 Google Ads API Usage

3.1 Purpose of API Access

The Google Ads API is used exclusively to retrieve performance reporting data for Google Ads accounts managed by i3Media under its MCC. No write operations of any kind are performed. The data retrieved is displayed within the i3Media reporting dashboard to assist staff in analysing campaign performance and preparing client reports.

3.4 GAQL Queries Executed

Account-level overview metrics (aggregated totals)

SELECT
  metrics.clicks,
  metrics.cost_micros,
  metrics.impressions,
  metrics.conversions,
  metrics.conversions_value
FROM customer
WHERE segments.date BETWEEN '{startDate}' AND '{endDate}'

Campaign-level performance breakdown

SELECT
  campaign.id,
  campaign.name,
  campaign.status,
  metrics.clicks,
  metrics.cost_micros,
  metrics.impressions,
  metrics.conversions,
  metrics.conversions_value
FROM campaign
WHERE segments.date BETWEEN '{startDate}' AND '{endDate}'
  AND campaign.status != 'REMOVED'
ORDER BY metrics.cost_micros DESC
LIMIT 20

Ad group-level performance breakdown

SELECT
  ad_group.id,
  ad_group.name,
  campaign.name,
  metrics.clicks,
  metrics.cost_micros,
  metrics.impressions,
  metrics.conversions,
  metrics.conversions_value
FROM ad_group
WHERE segments.date BETWEEN '{startDate}' AND '{endDate}'
  AND ad_group.status != 'REMOVED'
ORDER BY metrics.cost_micros DESC
LIMIT 20

Daily performance time series

SELECT
  segments.date,
  metrics.clicks,
  metrics.cost_micros,
  metrics.conversions,
  metrics.impressions
FROM customer
WHERE segments.date BETWEEN '{startDate}' AND '{endDate}'
ORDER BY segments.date ASC

Accessible customer accounts (MCC account listing)

SELECT
  customer_client.id,
  customer_client.descriptive_name,
  customer_client.currency_code,
  customer_client.manager
FROM customer_client
WHERE customer_client.status = 'ENABLED'
ORDER BY customer_client.descriptive_name ASC

4 Technical Architecture & Data Flow

4.1 High-Level Architecture

The tool is a server-rendered Next.js web application. All Google Ads API calls are made server-side from API route handlers — the browser never communicates directly with the Google Ads API, and credentials are never exposed to the client.

All API credentials (developer token, OAuth client ID/secret, refresh token) are stored as server-side environment variables. They are never sent to the browser.

4.2 Authentication Flow

  1. The application stores an OAuth2 refresh token obtained via a one-time offline OAuth consent flow using the https://www.googleapis.com/auth/adwords scope.
  2. For each API request, the server exchanges the refresh token for a short-lived access token by calling POST https://oauth2.googleapis.com/token.
  3. The access token is used to authenticate the Google Ads REST API call, along with the developer-token and login-customer-id (MCC account ID) headers.
  4. The access token exists only in server memory for the duration of the request and is never persisted or logged.

4.3 Request Lifecycle

  1. A logged-in i3Media staff member selects a client and date range in the dashboard.
  2. The browser sends a request to GET /api/google-ads?customerId=…&startDate=…&endDate=… — an internal Next.js API route.
  3. The server authenticates the staff user's session (JWT cookie).
  4. The server fetches a fresh OAuth2 access token using the stored refresh token.
  5. The server makes four parallel POST calls to the Google Ads REST API (overview, campaigns, ad groups, daily data).
  6. Results are formatted and returned as JSON to the browser.
  7. The dashboard displays formatted metrics (costs converted from micros, ROAS calculated, etc.).
  8. No data from step 6 is stored in the database — it is fetched fresh on every request.

5 Feature Description & UI Overview

5.1 Client Dashboard — Google Ads Tab

Each client in the system has a dedicated dashboard. The dashboard has tabs for each integrated service. The "Paid Search (Google Ads)" tab displays the following:

Summary Metric Cards

  • Clicks — total clicks in the selected date range
  • Cost — total spend (converted from micros to currency)
  • Conversions — total reported conversions
  • Conversion Value — total reported conversion value
  • ROAS — Return on Ad Spend (conversions_value ÷ cost)
  • CPA — Cost per Acquisition (cost ÷ conversions)
  • Impressions — total ad impressions
  • CTR — Click-Through Rate (clicks ÷ impressions)
  • Avg. CPC — Average Cost Per Click

Spend & Clicks Time Series Chart

A dual-axis area chart showing daily cost and click volume over the selected date range, allowing account managers to identify trends and anomalies.

Campaign Performance Table

A sortable table listing the top 20 campaigns by spend, with columns for clicks, cost, conversions, conversion value, ROAS, and CTR. ROAS is colour-coded (green ≥ 2×, amber ≥ 1×, red < 1×) for quick performance assessment.

Ad Group Performance Table

A table listing the top 20 ad groups by spend with their parent campaign name, clicks, cost, conversions, conversion value, and ROAS.

5.2 Client Settings — Account Selection

When configuring a client, the tool calls the Google Ads API to list all enabled customer accounts accessible under the i3Media MCC account. Staff select the appropriate customer ID from a dropdown. This selection is stored in the application database and used for subsequent API calls for that client.

5.3 Date Range Selection

Staff can select predefined date ranges (Last 7 days, 30 days, 90 days, 6 months) or a custom date range using a date picker. The selected dates are passed as segments.date BETWEEN filters in all GAQL queries.

6.1 Data Stored

Developer token Server environment variable Required API header
Campaign / metric data Not stored Data is fetched live on each request and returned to the browser; nothing is persisted

6.2 Data Not Stored

Performance data returned by the Google Ads API (clicks, costs, conversions, campaign names, ad content, etc.) is not stored in any database or cache. Each dashboard view fetches fresh data directly from the API. No Google Ads data is retained after the HTTP response is sent to the browser.

6.3 Access Controls

  • The application requires session authentication (email + password) for all pages and API routes.
  • Only i3Media staff with valid login credentials can access the tool.
  • The tool is not publicly hosted or accessible to clients.
  • All credentials are stored as server-side environment variables; they are never exposed in client-side JavaScript or HTML.

7 Security Practices

PracticeImplementation
Credential storage All API credentials stored as OS-level environment variables on the server. Never committed to version control (listed in .gitignore).
Token handling OAuth2 access tokens are obtained server-side per-request, exist only in memory, and are never logged or serialised.
HTTPS Application is served over HTTPS in production. All Google API calls use HTTPS.
Minimal scope OAuth consent is requested for only the adwords scope.
Read-only access No write operations are performed against any Google Ads account. All GAQL queries use SELECT only.
Session security User sessions use cryptographically signed JWT tokens with expiry. All API routes validate the session before processing.
Input validation All parameters (customer ID, dates) are validated server-side before being interpolated into GAQL queries. Date ranges are sanitised to prevent injection.

9 Features Not Used

The following Google Ads API capabilities are explicitly not used by this tool:

  • Campaign, ad group, ad, or keyword creation or modification
  • Budget management or billing services
  • Audience or remarketing list management
  • Keyword planning or idea generation
  • Account creation (CustomerService.CreateCustomerClient)
  • User management or access invitations
  • Automated bidding or bid strategy management
  • Conversion action creation or modification

10 Summary

The i3Media Client Reporting Tool uses the Google Ads API for a single, well-defined purpose: to retrieve performance metrics for Google Ads campaigns managed under the i3Media MCC account and display them to i3Media staff in a private internal dashboard.

All API access is read-only. No data is stored beyond the minimum required (customer ID per client). Credentials are handled securely server-side. The tool is not publicly accessible and is used exclusively by authorised i3Media employees.

Basic Access is required because the tool must query production accounts (live Google Ads accounts with real campaigns) and requires more than the 2,880 daily operations provided by Explorer Access in order to serve all managed client accounts across the i3Media portfolio with the 15,000 operations/day limit of Basic Access.
